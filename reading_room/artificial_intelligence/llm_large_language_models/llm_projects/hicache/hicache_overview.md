# HiCache 项目速览（MaaS 视角）

> 目标：帮助你在 10 分钟内理解 HiCache 的项目背景、能力边界、工程落地依赖、发展趋势与成熟度。

---

## 一句话定位

HiCache 是 SGLang 生态中的分层 KV Cache 管理能力：把原本主要驻留在 GPU HBM 的前缀缓存，扩展到 CPU 内存与外部存储（如 RDMA 内存池或分布式文件系统），以换取更低 TTFT 与更高吞吐，尤其适合长上下文、多轮对话、Agent 类负载。

---

## 1. 项目背景（Why Now）

在 MaaS 场景下，KV Cache 的矛盾通常同时出现：
- 容量不够：GPU HBM 无法承载更长上下文与更多并发会话。
- 带宽受限：Decode 阶段常受内存访问瓶颈影响。
- 复用不足：跨会话公共前缀复用率上不去，导致重复 Prefill 成本高。

HiCache 的核心出发点是：
- 在不强依赖 P-D 大规模重构的前提下，先把单实例内缓存层次做深。
- 让 Radix/Prefix 复用能力不只停留在 GPU 层，而是可延展到多层存储。
- 通过更细粒度的预取与写回策略，在延迟、吞吐、容量之间做可配置折中。

---

## 2. 项目能力边界（What It Can / Cannot Do）

### 2.1 擅长解决的问题

- 单实例内的 KV 容量扩展：
  从 GPU-only 扩展到 GPU + DRAM + 外部存储。
- 热前缀复用效率提升：
  对多轮会话、重复系统提示词、Agent 工具调用链更友好。
- TTFT 与吞吐平衡：
  通过预取策略和分层存储，降低长请求冷启动成本。
- 后端可插拔：
  可挂接不同存储后端，便于按成本/性能做工程取舍。

### 2.2 非目标与边界

- 不是完整的 MaaS 编排系统：
  HiCache 不负责 API 网关、鉴权、计费、路由编排。
- 不是 P-D 分离的替代品：
  它可以和 Mooncake/MemServe 等集群级架构叠加，而非二选一。
- 不能消除模型本身计算成本：
  主要优化 KV 相关内存与 I/O 路径，不改变模型 FLOPs 本质。
- 对工作负载分布敏感：
  若请求高度随机、前缀复用弱，收益会明显下降。

### 2.3 归类建议（便于和现有图谱对齐）

- 更准确归类：A x C 融合能力
- A：框架内前缀复用能力（Radix/Prefix）
- C：分层存储扩展能力（GPU/DRAM/外部存储）
- 与 B（Disaggregated KV）关系：可组合，不互斥

## 3. 工程落地上下文依赖（How To Land）

下面按“必须具备”和“建议具备”划分。

### 3.1 必须具备

- 推理框架与版本基线：
  需要与 SGLang/推理引擎版本保持兼容，先固定可复现实验版本。
- 可观测性基础：
  至少要有 TTFT、TPOT、tokens/s、KV 命中率、I/O 等待时间、GPU 利用率。
- 存储后端可用性：
  本地 NVMe、远程内存池、或分布式文件系统至少一种稳定可用。
- 容量规划：
  清晰定义会话数、上下文长度、峰值并发，反推各层缓存配额。

### 3.2 建议具备

- 网络与 NUMA 调优能力：
  涉及 RDMA/跨节点存储时，网络与内存拓扑会直接决定收益上限。
- 分层策略灰度机制：
  支持按模型、租户、路由策略做渐进放量和回滚。
- 负载画像体系：
  区分对话类、Agent 类、批处理类流量，分别调预取与写回策略。
- 故障降级路径：
  外部存储抖动时要能自动回退到更保守策略，避免全链路雪崩。

### 3.3 常见落地路径（推荐顺序）

1. 单机验证：先只开 GPU + DRAM 分层，确认收益方向与稳定性。
2. 小流量灰度：引入外部存储后端，观察 TTFT 尾延迟与抖动。
3. 分租户策略：为高复用业务优先启用更激进的预取策略。
4. 与集群级 KV 架构叠加：再评估是否接入 B 类能力（如 P-D 分离）。

### 3.4 评估指标建议（最小闭环）

- 效率：TTFT p50/p95、TPOT、整体吞吐 tokens/s
- 成本：单位请求 GPU 秒数、外部存储 I/O 成本、网络成本
- 稳定性：超时率、重试率、尾延迟抖动
- 复用质量：前缀命中率、跨会话命中率、有效卸载比

## 4. 发展趋势与成熟度（As of 2026-05）

### 4.1 趋势判断

- 趋势 1：单实例分层缓存会成为默认能力
  原因：长上下文和 Agent 负载已成主流，GPU-only 缓存难以支撑成本与容量目标。

- 趋势 2：与集群级 KV 池叠加成为主流架构
  形态：实例内分层（HiCache 类）+ 集群级 KV 解耦（Mooncake/MemServe 类）。

- 趋势 3：策略层会从“固定阈值”走向“负载自适应”
  预取、写回、淘汰策略会更多依赖实时画像与在线反馈控制。

- 趋势 4：生态接口趋于标准化
  后端适配接口、观测指标、压测方法会逐步形成事实标准。

### 4.2 成熟度评估（工程视角）

- 技术成熟度：中高
  核心思路与收益方向已经被多方验证，具备实战价值。

- 产品成熟度：中等
  在不同组织、不同后端上的开箱一致性仍取决于工程集成质量。

- 组织成熟度要求：中高
  想稳定吃到收益，需要推理平台、存储、网络、SRE 协同，而非单团队可独立完成。

### 4.3 采用建议

- 适合优先采用：
  长上下文、多轮对话、Agent 任务、复用率较高业务。

- 谨慎采用：
  短上下文、一次性请求、复用率低、存储/网络基础薄弱场景。

## 5. 快速决策清单（给负责人）

上线前至少回答这 8 个问题：

1. 我们的核心收益目标是 TTFT、吞吐，还是 GPU 成本下降？
2. 目标业务的前缀复用率是否足够高？
3. 现有可观测性能否定位瓶颈在 KV/I-O 而非纯算力？
4. DRAM/NVMe/RDMA 哪一层是当前最现实的扩展路径？
5. 预取和写回策略是否支持按租户或路由灰度？
6. 外部后端抖动时的降级与回滚路径是否明确？
7. 我们是否需要与 B 类架构叠加，而不是替代？
8. 是否有统一压测集用于持续回归（含尾延迟）？

若以上问题中有 4 项以上无法回答，建议先做 2 周基线压测，再推进生产灰度。
