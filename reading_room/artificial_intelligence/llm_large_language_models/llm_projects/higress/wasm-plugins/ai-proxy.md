# AI Proxy 插件实现原理

## 1. 插件定位

`ai-proxy` 是部署在 Higress 网关数据面中的 WASM 插件，核心目标是把“多模型、多供应商、多协议差异”收敛成统一入口。

对业务方来说，调用方只面向一个网关 API；对平台方来说，插件在网关层完成模型路由、鉴权注入、协议适配、流式透传和可观测采集。

## 2. 架构分层

从实现职责看，`ai-proxy` 可以拆成 4 层：

1. `Request Normalize`：统一输入格式，解析模型名、参数、流式标记。
2. `Route & Policy`：根据路由规则选择上游模型服务，并应用限流/超时/重试策略。
3. `Provider Adapter`：对接不同模型提供方时做请求/响应字段映射。
4. `Stream & Observe`：处理 SSE/分块响应透传，同时记录指标与日志。

## 3. 请求处理主流程

一个典型请求在插件中的生命周期通常如下：

1. 进入网关后，插件从 Header/Body 中提取模型目标、租户标识和调用参数。
2. 做前置校验：鉴权信息、必填参数、请求大小、速率限制等。
3. 根据配置完成路由决策：
	- 固定路由：模型名 -> 固定上游。
	- 策略路由：按租户、标签、权重、灰度规则分流。
4. 在转发前进行请求重写：
	- 注入上游 API Key/Token。
	- 协议字段映射（例如不同供应商的模型字段和参数命名差异）。
5. 将请求转发到目标推理服务（如 OpenAI-compatible API、云厂商模型网关、私有推理服务）。
6. 接收响应并统一输出：
	- 非流式：聚合后返回统一 JSON。
	- 流式：逐段透传，尽量避免额外缓冲导致时延放大。
7. 在请求结束时写入观测数据：延迟、状态码、token 用量（若可得）、上游实例信息。

## 4. 流式返回实现要点

LLM 场景里，流式体验通常比吞吐更敏感。`ai-proxy` 的关键原则是“尽早转发、最少改写”：

- 尽量按 chunk 透传，减少中间缓存。
- 在必要场景才做增量改写（例如统一事件格式）。
- 当上游中断时，优先输出可识别的错误事件，避免客户端无感超时。

## 5. 协议适配机制

不同模型服务常见差异：

- 请求字段名不同（`model`/`engine`/`deployment`）。
- 采样参数命名或范围不同（`temperature`、`top_p`、`max_tokens`）。
- 流式事件格式不同（SSE 事件字段、结束标记）。

`ai-proxy` 的实现思路是通过“适配层”把这些差异封装在网关侧，业务调用保持稳定接口，减少客户端 SDK 分叉。

## 6. 容错与稳定性策略

典型稳定性策略包括：

- 超时控制：区分连接超时与请求超时。
- 重试策略：仅在幂等或可安全重试的场景触发。
- 熔断与降级：上游异常持续时，快速失败或切换候选上游。
- 限流与配额：在网关层保护后端推理集群。

## 7. 可观测与运维关注点

建议重点关注这些指标：

- 请求成功率（按模型、租户、上游维度）。
- P95/P99 延迟（区分首 token 延迟与全响应延迟）。
- 上游错误分布（4xx/5xx/超时/连接失败）。
- 流式中断率与重试命中率。

这些数据可直接用于容量规划、路由策略优化和异常回溯。

## 8. 与推理平台集成建议

在 LLM 平台落地时，可以把 `ai-proxy` 作为“模型网关层”能力：

- 北向提供统一 API（对应用稳定）。
- 南向对接多种推理后端（云上/自建混合）。
- 中间通过策略配置实现灰度、A/B、按租户治理。

这样可以把模型侧变化与业务侧调用解耦，提升平台演进速度。
